# AGENTS.md — Verstak Mobile Remote Access

Этот файл — постоянный контекст и handoff для любых AI-агентов, которые продолжают разработку мобильного доступа Verstak.

Перед работой в `mobile/` и `electron/mobile-bridge/` обязательно прочитать:

1. Этот файл.
2. `docs/superpowers/specs/2026-07-11-mobile-remote-access-design.md`.
3. `docs/superpowers/plans/2026-07-11-mobile-remote-access.md`.
4. `mobile/README.md`.
5. Корневой `AGENTS.md` проекта Verstak.

## 1. Что мы делаем

Создаём единый доступ к настольному Verstak с телефона по модели ChatGPT/Codex:

- пользователь открывает мобильный интерфейс;
- выбирает свой компьютер;
- выбирает разрешённый проект или папку;
- создаёт новый чат или продолжает существующий;
- пишет, диктует или прикрепляет задачу;
- задача выполняется локальным Verstak на компьютере;
- телефон показывает ход выполнения, вопросы, подтверждения и результат.

Это не удалённый рабочий стол. Экран компьютера, мышь и произвольный терминал не транслируются.

Главный принцип архитектуры:

`Телефон -> защищённый relay -> desktop Verstak -> локальные файлы и AI-инструменты`.

Компьютер остаётся единственным исполнителем и источником истины. Relay только передаёт доменные команды и события.

## 2. Согласованный продуктовый MVP

В целевом первом контуре должны быть:

1. Вход в мобильный Verstak.
2. Список подключённых компьютеров и online/offline статус.
3. Выбор компьютера.
4. Просмотр только разрешённых проектов и папок.
5. Выбор папки и создание отдельного чата из неё.
6. Продолжение существующего чата.
7. Текстовый и голосовой ввод.
8. Прикрепление файла с телефона.
9. Локальное выполнение задачи desktop Verstak.
10. Поток ответа и статусов инструментов.
11. Подтверждение или отклонение опасных операций.
12. Остановка запуска.
13. Просмотр изменённых файлов и unified diff.

Первая клиентская форма — PWA. Нативные приложения Android/iOS и публикация в магазинах не входят в текущий этап.

## 6. Ближайший порядок работ

Не начинать с production-развёртывания. Следовать этому порядку:

### Этап A — реальная локальная проверка

- запустить relay, desktop и PWA по `mobile/README.md`;
- открыть PWA на телефоне в одной Wi-Fi сети;
- выбрать проект;
- создать чат;
- отправить безопасную задачу на чтение;
- отправить задачу с тестовым изменением файла;
- убедиться, что результат появился на компьютере и в истории телефона;
- проверить stop и reconnect.

### Этап B — закрыть mobile control loop

- довести поток desktop `ai:event` до полноценной замены polling;
- добавить approvals;
- добавить changes/diff;
- завершить attachment upload;
- покрыть каждый сценарий integration-тестом.

### Этап C — доступ через интернет

- выбрать инфраструктуру relay;
- добавить production auth и device pairing;
- настроить TLS, rate limits, TTL очереди и отзыв устройств;
- провести security review;
- только потом выполнить controlled deployment.

### Этап D — упаковка

- проверить PWA install flow на Android и iPhone;
- добавить push notifications;
- после проверки спроса решить, нужна ли нативная оболочка.

## 7. Границы безопасности

Нельзя:

- передавать телефону полный доступ к диску;
- принимать абсолютный filesystem path как доверенный аргумент;
- добавлять универсальный `shell.run` в публичный мобильный протокол;
- хранить provider keys, prompts, содержимое файлов и tool results в relay-логах;
- коммитить relay token, credentials или pairing credential;
- автоматически повторять approval после reconnect;
- развёртывать relay в production без явного разрешения Павла;
- менять существующие renderer IPC-контракты без синхронного обновления `preload.ts` и `src/types/api.d.ts`;
- сливать feature-ветку в `main` до ручного smoke на телефоне.

Разрешённые папки определяются только desktop-стороной. Любой мобильный путь должен проходить через `rootId`, `safeRealJoin` и forbidden-path policy.

## 8. Критерий готовности первого релиза

Первый релиз готов, когда с телефона через интернет можно:

1. Авторизоваться и увидеть свой online-компьютер.
2. Выбрать разрешённую папку.
3. Создать чат и отправить задачу.
4. Увидеть поток выполнения без ручного обновления.
5. Подтвердить или отклонить опасное действие.
6. Остановить задачу.
7. Просмотреть изменения файлов.
8. Прикрепить файл в выбранный проект.
9. Переподключиться без повторного запуска команды.

При этом файлы, ключи, SQLite и выполнение должны оставаться на компьютере, а relay не должен становиться облачным хранилищем проекта.
