# Гибридный оркестратор API ↔ CLI — проектирование

> Детальное проектирование поверх концептуального ТЗ (см. чат-источник).
> Главный вывод: **~75% концепта уже реализовано.** Этот документ отделяет готовое от недостающего и фиксирует архитектурное решение по недостающему.

## 1. Концепт ↔ реальность (сверено с кодом)

Из 4 механизмов ТЗ три уже в проде:

| Механизм ТЗ | Статус | Где в коде |
|---|---|---|
| Единый лог истории (SQLite) | ✅ готов | `storage/chats.ts`, `chat-sessions.ts` — CLI-ответ пишется как обычное `assistant`-сообщение; tool-события идут в UI параллельно |
| Адаптивный сериализатор `buildCliPrompt` | ✅ готов | `ai/cli-prompt.ts:60` — stdin = system+user-layer + context-pack + skill-layer + история + verify-hint + текущий запрос |
| Синхронизация состояния файлов | ✅ готов | `ai/context-pack.ts` — пересборка на каждый `ai:send`: git-status + recent-writes + project-map |
| Авто-роутинг API↔CLI | 🟡 частично (Фаза 1) | `ai/smart-router.ts` — был только выбор модели внутри API-провайдера; добавлен `detectCliWorthiness` (детектор терминальных задач) |

Сверх ТЗ уже есть:
- **Ручное переключение провайдера mid-chat** (Сценарий А) — провайдер+модель per-chat (`chat_sessions.provider_id/model`, `setProviderModel`), история provider-agnostic. **Технически работает; нужна живая проверка UI-проводки.**
- **`delegate_task` / мультиагент** — под-агент крутит собственный agent-loop с инструментами, результат как `tool_result`. `delegate_parallel` **спроектирован смешивать API и CLI в одном батче** (`tool-handlers.ts:1105`). Это и есть транспортный фундамент Сценария Б.

## 2. Что реально отсутствовало

1. **Сигнал роутинга** — система не подсказывала, что задача требует терминального цикла (сборка/тесты/итеративная отладка), которые автономнее делает CLI.
2. **Честность по безопасности.** ТЗ декларирует «denylist одинаково строг к CLI». **По факту CLI-провайдеры обходят** `path-policy`/`secret-scanner`/`command-policy` Verstak — они крутят собственный loop и полагаются на свой sandbox (Claude Code v2, Codex native). Это inherent-ограничение CLI, его надо признавать, а не обещать обратное.

## 3. Архитектурное решение

**Форма авто-роутинга — подсказка, а не молчаливый свитч.** Причина принципиальная: ядро Verstak — *контроль и прозрачность*. Молчаливая передача дорогого CLI-прогона без ведома пользователя противоречит самой ценности продукта и риску расхода токенов (DoD ТЗ: «не запускать дорогой CLI там, где достаточно ответа»). Поэтому:

- **Детектор** (`detectCliWorthiness`) — чистая функция: терминальная ли задача. Покрыта тестами.
- **Подсказка** — когда активен API-провайдер и задача терминальная, эмитим `info`-событие: «похоже на терминальную задачу → автономнее справится CLI». Пользователь решает: переключить провайдер (Сценарий А) или попросить делегировать шаг на CLI (Сценарий Б через `delegate_task`).
- **Транспорт Сценария Б** — переиспользуем `delegate_task` с CLI-провайдером: API-агент делегирует тяжёлый терминальный шаг на Claude Code, тот крутит свой loop, результат возвращается как `tool_result` и виден в Timeline (прозрачность DoD соблюдена).

## 4. Фазы

### Фаза 1 — ГОТОВО (эта сессия)
- `smart-router.ts`: `detectCliWorthiness(messages)` — детектор терминальных задач (сборка/типы/тесты/отладка/локальное окружение), RU+EN. 7 тестов.
- `ai.ts`: эмиссия honest info-подсказки при API-провайдере + терминальной задаче. Поведение не меняется (только подсказка). type ✓ · build ✓.

### Фаза 2 — UI-подсказка (зона Ильи + живая проверка)
- Pill в Timeline/composer: «🔧 Терминальная задача → Claude Code?» с кнопкой в один клик (переключает `setProviderModel` на сконфигурированный CLI).
- Гейт показа: подсказывать конкретный CLI только если он **сконфигурирован/авторизован** (иначе не называть инструмент, которого нет).
- Живая проверка Сценария А end-to-end: переключение API→CLI→API в одном чате без потери нити.

### Фаза 3 — Делегирование на CLI (Сценарий Б, прозрачный путь) — ТРАНСПОРТ ГОТОВ
Транспорт проверен по коду и зафиксирован тестами (`tests/ai/sub-agent-cli-delegate.test.ts`):
- `buildSubCreateOptions` (`tool-handlers.ts:786`) строит опции для `claude-cli` — oauth-токен (headless+Max) + cwd проекта.
- `claude-cli.send` (`claude-cli.ts:102`) сам сериализует `messages` через `buildCliPrompt` в stdin; API-tools/results игнорирует (CLI крутит свой loop).
- `runSubAgentLoop` корректно обрабатывает CLI-стиль ответа: текст без `tool-call` → `completed` с текстом, Verstak-хендлеры НЕ дёргаются; usage CLI учитывается в cost-guard сессии (суб не обходит лимит).
- **Остаток:** живой прогон реального CLI-бинаря (нужен сконфигурированный Claude Code) — единственное, что нельзя подтвердить без живого окружения.
- Опционально (отдельно): в описание `delegate_task` добавить, что для терминальных шагов можно таргетить CLI-провайдер — чтобы модель сама предлагала делегирование. Это поведенческое изменение, делать после Фазы 2 (UI-pill) и живой проверки.

### Не делаем
- **Молчаливый полный авто-свитч чата на CLI** — против ценности контроля; отвергнуто осознанно.
- **Дублирование denylist поверх CLI** — CLI крутит свой loop, перехватить его файловые операции нельзя; полагаемся на native sandbox CLI и честно это документируем.

## 5. Критерии успеха (ревизия DoD ТЗ)

| DoD ТЗ | Как закрывается |
|---|---|
| Контекст не теряется при API↔CLI↔API | ✅ единый лог + `buildCliPrompt` + context-pack (готово); ⏳ живая проверка — Фаза 2 |
| Прозрачность запуска CLI (стрим, не чёрный ящик) | ✅ CLI стримит stream-json в чат; делегирование видно в Timeline как subagent-run |
| Безопасность одинаково строга к CLI | 🟡 **честная формулировка:** API-тулзы гейтятся Verstak; CLI полагается на собственный sandbox — задекларировано, не обещаем невозможное |
| Экономия токенов (не запускать дорогой CLI зря) | ✅ детектор не триггерит на простых вопросах (тесты это проверяют); подсказка, а не авто-запуск |
