# Verstak Mobile Remote Access — дизайн

## Цель

Дать пользователю единый чат между телефоном и настольным Verstak: выбрать подключённый компьютер и разрешённую папку, создать или продолжить чат, надиктовать/отправить задачу, наблюдать локальное выполнение, подтверждать опасные действия, останавливать запуск и просматривать изменённые файлы.

## Продуктовая граница MVP

- Телефон не транслирует экран и не управляет мышью.
- Настольный Verstak остаётся единственным исполнителем: файлы, терминал, провайдеры, ключи и SQLite остаются на компьютере.
- Телефон показывает только явно разрешённые в desktop-профиле проекты и папки.
- Компьютер должен быть включён, Verstak запущен и иметь соединение с relay.
- Первая клиентская форма — устанавливаемая PWA; нативные оболочки Android/iOS не входят в MVP.
- Внешнее production-развёртывание relay и настройка домена не входят в локальную реализацию без отдельного разрешения.

## Архитектура

Система состоит из трёх изолированных узлов:

1. `mobile/app` — mobile-first PWA. Авторизация, список компьютеров, разрешённые корни, проекты, чаты, composer, поток событий, approvals, stop и diff.
2. `mobile/relay` — слепой транспорт. Он аутентифицирует устройства, маршрутизирует команды и события по `accountId/deviceId`, хранит только короткоживущие недоставленные конверты и не получает ключи провайдеров или произвольный доступ к диску.
3. `electron/mobile-bridge` — desktop-клиент relay. Он публикует безопасный снимок доступных проектов/папок и переводит мобильные команды в существующие сервисы чатов и AI-запусков.

Основной поток:

`PWA -> HTTPS/WSS relay -> исходящее WSS-соединение desktop bridge -> локальные сервисы Verstak`.

Исходящее соединение позволяет работать из любой сети без IP-адресов, проброса портов и Tailscale.

## Идентификация и сопряжение

- Пользователь входит в PWA по аккаунту Verstak.
- Desktop создаёт одноразовый pairing-код с TTL 10 минут.
- После подтверждения desktop получает отдельный device credential, хранимый через `safeStorage`.
- Credential можно отозвать на компьютере; relay принимает сообщения только в пределах владельца устройства.
- Логи и сетевые payload проходят существующую редакцию секретов.

Для локального MVP аккаунт и relay credential задаются через локальную конфигурацию разработки; production OAuth не имитируется.

## Разрешённые папки

- Bridge не публикует весь диск.
- В MVP доступны только зарегистрированные проекты Verstak и отдельно добавленные корни.
- В каждом запросе используется стабильный `rootId`; абсолютный путь никогда не приходит от телефона как доверенный аргумент.
- Относительные пути проходят `safeRealJoin` и существующий forbidden-path policy.
- Файл с телефона сначала попадает в staging, затем записывается только после проверки размера, имени, разрешённого root и явного назначения.

## Протокол

Все сообщения — versioned envelopes:

```ts
type MobileEnvelope = {
  v: 1
  id: string
  accountId: string
  deviceId: string
  kind: string
  sentAt: number
  payload: unknown
}
```

Команды MVP: `device.snapshot`, `roots.list`, `files.list`, `chats.list`, `chat.create`, `chat.history`, `chat.send`, `run.stop`, `approval.resolve`, `changes.list`, `attachment.stage`.

События MVP: `device.status`, `chat.message`, `run.status`, `run.event`, `approval.requested`, `changes.updated`, `command.error`.

Каждая команда имеет `id`; desktop сохраняет короткий idempotency-cache, чтобы повторная доставка не запускала задачу дважды. Destructive/confirm-write действия никогда не подтверждаются автоматически после reconnect.

## Состояние и восстановление

- Источник истины по чатам, сообщениям, проектам и запускам — desktop SQLite.
- Relay хранит только присутствие устройств и короткоживущую очередь конвертов.
- PWA после reconnect запрашивает свежий snapshot и историю, а не пытается восстановить истину из локального кэша.
- Если desktop offline, создание запуска блокируется понятным статусом; текст задачи можно сохранить только как локальный черновик телефона.

## Безопасность

- Только TLS/WSS вне localhost.
- Relay не принимает filesystem path и shell command как публичные универсальные операции.
- Мобильный API строится на белом списке доменных команд.
- Опасные действия используют текущую mode-policy/approval-модель desktop.
- Pairing-коды одноразовые; device credentials ротируются и отзываются.
- Rate limits применяются к login, pairing, attachment и command streams.
- В relay-логах запрещены содержимое файлов, prompts, ключи и tool results.

## UX MVP

1. Экран входа/локальной dev-сессии.
2. Список компьютеров со статусом online/offline и временем последнего контакта.
3. Экран компьютера: разрешённые проекты и папки.
4. Экран папки: дерево файлов и список чатов.
5. Чат: текст, диктовка средствами браузера, attachment picker, поток ответа и статусы инструментов.
6. Approval-card с явными «Разрешить» / «Отклонить».
7. Changes-view: список файлов и unified diff только для чтения.

## Ошибки

- Offline desktop: команда не исполняется, UI сохраняет черновик.
- Потеря relay: exponential reconnect с jitter и snapshot после восстановления.
- Дубликат команды: возвращается исходный результат по envelope id.
- Истёкший pairing: код отклоняется без раскрытия существования аккаунта/устройства.
- Недоступная папка: root помечается unavailable; путь наружу не раскрывается.
- Слишком большой attachment: отклонение до передачи desktop.

## Проверка

- Unit: schema/envelope validation, root capability mapping, idempotency, pairing TTL, redaction.
- Integration: relay маршрутизирует только владельцу; offline/reconnect; duplicate `chat.send`; approval не переигрывается.
- Desktop: адаптер существующих chats/AI events без изменения контрактов renderer IPC.
- PWA: device -> root -> chat -> send -> stream -> stop и approval сценарии.
- Security: traversal, forbidden paths, oversized attachment, чужой account/device, replay pairing.
- Финальный ручной smoke на телефоне и компьютере выполняется вместе с пользователем после его возвращения.

## Не входит в MVP

- Удалённый экран, терминал общего назначения и произвольные shell-команды.
- Работа при выключенном компьютере.
- Облачное хранение репозиториев, истории чатов или provider credentials.
- App Store / Google Play публикация.
- Совместная работа нескольких пользователей над одним устройством.
