## 1. Несущая конструкция (сломать = сломать всё)

**Диспетчер `ai.onEvent` (E18, строки 1327–1753, 427 строк) — замыкание ПЕРВОГО рендера.**

```
}, [updateLastAssistant, setStreaming])   // обе — стабильные zustand-экшены
```

Подписка ставится **ровно один раз за маунт**. Свежесть данных достигается ДВУМЯ механизмами и
только ими:
1. `useProject.getState()` — снимок стора на момент события (не реактивный);
2. **refs**.

**Ловушка №1 при извлечении.** Любая попытка передать в хук реактивные значения через deps
(`provider`, `activeChatId`, `helpChatId`, `appliedSkills`…) заставит подписку пересоздаваться на
каждом рендере. Между `off()` и `onEvent()` события теряются молча: текст ответа пропадает, `done`
не приходит → **`isStreaming` залипает навсегда**.

**Ловушка №2.** Обратное тоже верно: оставить deps пустыми, но вынести хелперы наружу — и они
потеряют доступ к тем самым ref'ам, которые компенсируют устаревание.

**Ловушка №3.** `trackPersistedAssistantEvent(id, event)` вызывается **первой строкой, до любого
роутинга** (1329). Это единственный путь персиста стриминга для фоновых чатов. Перенос вызова
внутрь веток = ответ виден на экране, но исчезает после перезапуска.

## 2. Порядок веток роутера несёт инварианты (каждый уже был багом)

Цепочка ранних `return` (1338–1471) — **маршрутизация идёт по OWNER'у, и только потом по типу события**:

```
help-owner (1338) → review-owner (1352) → chat-owner в ЧУЖОМ проекте (1372, с исключением
plan-approval на 1376) → фоновый проект (1407) → plan-approval глобально (1427) →
снятие pendingPlan на done/error (1433) → фоновый чат/help (1439) → if (!owner) return (1471) →
активный чат (1472+)
```

| Инвариант | Что сломается при «нормализации» |
|---|---|
| исключение `plan-approval` на 1376 | модалка плана перестанет всплывать глобально → агент фонового чата **навсегда зависнет** в await plan-gate (тихий дедлок) |
| `if (!owner) return` стоит ПОСЛЕ снятия pendingPlan (1433) | «сначала отсеем неизвестный sendId» → **ghost-модалка** плана после stop() |
| chat-owner (1372) опережает фоновый проект (1407) | события чата уйдут в `applyEventToSession`, который **НЕ пишет в БД** → тихая потеря ответа при reload |

Наивная декомпозиция «по типу события» (switch/таблица обработчиков) разрушает все три.

## 3. Найденные ЛАТЕНТНЫЕ дефекты (не вносить в новый код, чинить осознанно)

### 3.1 `helpChatId` в замыкании роутера навсегда `null` — ПОДТВЕРЖДЁН

- `helpChatId: null` в store init (`projectStore.ts:323`), выставляется только при открытии
  справки (`projectStore.ts:1120`).
- Роутер — mount-замыкание → `pendingScopeKeyFor` (1073) внутри него навсегда видит `null` →
  для help-owner возвращает `help:global`, тогда как живой `pendingScopeKey` (945) — `help:<реальный id>`.
- Итог: `clearPendingSupplementsForScope` чистит несуществующий scope; `flushQueuedForOwner` уходит
  в ветку «чужой scope» → `startQueuedBackgroundChatMessage` возвращает false для isHelp → элемент
  кладётся обратно в очередь `help:global`.
- **Сейчас замаскировано** страховочным эффектом 2439 (флаш по `isStreaming`/`queuedMessages.length`/`pendingScopeKey`).

⚠️ **Извлечение очереди в модуль с «уборкой дублирующего эффекта 2439» немедленно проявит баг:**
очередь help-чата перестанет отправляться, плашка supplements залипнет.

### 3.2 `currentSendIdRef` — один слот на все чаты

Модель «один активный прогон» противоречит мульти-чат стримам, которые файл же поддерживает.
Чат A стримит → переключились в B → отправили → ref перезаписан sendId B. Stop останавливает
только B; прогон A живёт без UI-контроля. `appendTextToCurrentContext` (2480–2522) отправит
дополнение в **чужой прогон**.

→ Извлечение «context supplements» ОБЯЗАНО заменить ref на map `sendId по chatId`, иначе дефект
переедет дословно и станет заметнее.

### 3.3 `onSendChatReminder` (2883–2978) — вторая копия логики отправки

Дублирует ~70% `startQueuedBackgroundChatMessage`, но с **другим** набором перепроверок активности:
там `isActiveTarget` считается один раз (2906) и переиспользуется после трёх await'ов — гонка
«переключил чат во время await» **не закрыта**, в отличие от 2309–2311. При объединении в общий
модуль расхождение решать осознанно (по более строгому варианту), а не «слить как одинаковое».

## 4. Refs, присваиваемые ВО ВРЕМЯ рендера (линтер-ориентированный рефактор их сломает)

| Место | Что сломается при переносе в useEffect |
|---|---|
| `composerInputRef`/`AttachmentsRef`/`AppliedSkillsRef` (848–851) | E10 (862) сохраняет ПРЕДЫДУЩИЙ черновик по prevKey — он читает ref'ы, которые на рендере смены чата ещё содержат старый текст. Перенос в эффект → черновик старого чата запишется под ключ нового (**перетрёт его**) либо потеряется |
| `flushQueueRef.current` (2437) | `stop()` (3009) зовёт через ref → всегда свежая версия. Перенос в useEffect ломает stop(), вызванный в том же коммите, что и рендер |

Пара **E10 (загрузка черновика, 862) → E12 (autosave debounce 220 мс, 897)**: порядок объявления
обязателен. E12 читает `composerDraftKeyRef`, который мутирует E10. Разнести = черновик уедет в чужой чат.
У E12 cleanup отменяет таймер, но **не флашит** → последние ≤220 мс ввода теряются при unmount.

## 6. Страховочная сетка: чем можно и чем нельзя

**Нельзя сегодня:** снять baseline event trace с ЖИВОГО роутера. Он внутри React-компонента
(замыкания на refs/state), а в проекте `environment: 'node'`, **нет jsdom и testing-library**,
компоненты не рендерились в тестах никогда. Chat.tsx: 51 импорт, 18 неймспейсов `window.api`.

**Развилка (решение за координатором/Павлом):**

1. **Вложиться в сетку** — devDeps `jsdom` + `@testing-library/react`, `environmentMatchGlobs`
   (только `tests/components/*` в jsdom, остальное остаётся node/быстрым), фабрика мока
   `window.api` (18 неймспейсов), рендер Chat, прогон событий через захваченный `onEvent`-хэндлер.
   Дорого один раз — но в 2.0.9 **четыре** Chat.tsx-хирургии (route-changed пилюля, pinned-бейдж,
   fork/edit-rerun, rewind), и каждой эта сетка нужна.
2. **Сузить A** — карта (готова) + characterization на детерминированной зоне (очередь 1 таблицы
   выше, тесты пишутся сегодня без новых зависимостей) + каждую Chat.tsx-хирургию делать
   минимальным дифом с адверсариальным ревью. Дешевле, но самые рискованные срезы идут без сетки.

**Рекомендация:** вариант 1. План сам называет 2.0.9 самым рискованным релизом, а A — фундаментом.
Сетка окупается на первой же хирургии.
