# План исправления кодировки в настройках проекта

Дата: 2026-07-16  
Зона проблемы: `src/components/ProjectSettings.tsx`

## Что происходит

В окне настроек проекта слетели не шрифты, а кодировка русских строк. На скриншоте видны строки вида `РџР°СЂР°Рј...` вместо нормального русского текста. Это mojibake: русский UTF-8 текст однажды был прочитан или записан как Windows-1251, после чего уже повреждённый текст попал в исходник.

Динамические данные отображаются нормально:

- название проекта `Антис Студио`
- даты
- путь проекта
- часть данных из базы

Ломается именно статичный текст интерфейса:

- заголовки
- подписи блоков
- placeholder в полях
- кнопки
- confirm/alert сообщения
- aria-label/title

По текущей проверке в `src/components/ProjectSettings.tsx` найдено минимум 86 строк с mojibake.

## Почему в других окнах нормально

Большинство других экранов берут русский текст из `src/i18n/ru.ts`, где строки сейчас сохранены нормально. В `ProjectSettings.tsx` много текста зашито прямо в компоненте, минуя i18n. Поэтому при неудачном мердже или правке через Windows-консоль именно этот компонент оказался уязвим.

CSS здесь не главная причина. В `src/styles/layout.css` уже есть блок `Project settings typography guard`, который чинит font-family, жирность и uppercase-стиль. Но CSS не может исправить текст, который уже записан в исходнике в неправильной кодировке.

## Вероятная причина

Один из прошлых циклов правок был сделан через инструмент или команду, которая прочитала UTF-8 как Windows-1251 или записала файл обратно не в той кодировке. На Windows это легко получить через PowerShell/консольный вывод, если не следить за UTF-8.

Характерный пример:

```text
РџР°СЂР°РјРµС‚СЂС‹ РїСЂРѕРµРєС‚Р°
```

Восстанавливается в:

```text
Параметры проекта
```

Технически это подтверждается преобразованием:

```js
iconv.decode(iconv.encode(badText, 'win1251'), 'utf8')
```

## Цель исправления

1. Вернуть все русские строки в настройках проекта в нормальный UTF-8.
2. Убрать причину повторного появления проблемы.
3. Добавить автоматическую проверку, чтобы mojibake больше не попадал в коммит.
4. По возможности перевести статичный текст настроек проекта в i18n, как в остальных окнах.

## План работ

### 1. Зафиксировать текущий объём проблемы

Составить список всех повреждённых строк в `src/components/ProjectSettings.tsx`.

Минимальный критерий поиска:

- `Рџ`
- `РЎ`
- `Рњ`
- `Рќ`
- `Р°`
- `СЊ`
- `С‚`
- `СЃ`
- `вЂ`
- `Г—` там, где ожидался символ закрытия или стрелка

Важно: не заменять весь файл вслепую. В проекте могут быть легитимные русские строки в других местах, а также исторические fallback-строки для старых данных.

### 2. Восстановить текст в `ProjectSettings.tsx`

Вариант исправления:

1. Автоматически преобразовать только строковые литералы и JSX-текст, где найден mojibake.
2. Использовать `iconv-lite`:

```js
const fixed = iconv.decode(iconv.encode(bad, 'win1251'), 'utf8')
```

3. После автоматического восстановления вручную пройтись по тексту и привести формулировки к нормальному стилю Verstak:

- коротко
- без лишних точек в конце компактных подписей
- без жирного моно-шрифта
- понятным языком для пользователя

Особенно проверить:

- заголовок окна
- блок "Заметки"
- блок "Сведения"
- блок "Ярлыки"
- блок "Группа"
- блок "Расположение"
- блок "Статус проекта"
- блок "Цвет"
- блок "Удалённый проект"
- блок "Данные проекта"
- блок "Управление проектом"
- нижнюю панель "Изменений нет / Закрыть / Сохранить"
- confirm/alert перед очисткой, архивом и удалением

### 3. Вынести статичный текст в i18n

Чтобы проблема не повторялась, лучше убрать большую часть русских литералов из `ProjectSettings.tsx`.

Нужно добавить в `src/i18n/ru.ts` и `src/i18n/en.ts` отдельный раздел, например:

```ts
projectSettingsPanel: {
  title: string
  subtitle: string
  notesTitle: string
  notesDescription: string
  ...
}
```

После этого компонент должен брать текст через `useT()`, как остальные основные окна.

Почему так надёжнее:

- меньше русских строк внутри TSX
- легче проверять переводы одним местом
- меньше риск, что при правке компонента Windows-консоль испортит кириллицу
- проще поддерживать одинаковый стиль текста

### 4. Добавить guard от mojibake

Добавить скрипт, например:

```text
scripts/check-mojibake.cjs
```

Он должен сканировать:

- `src/**/*.ts`
- `src/**/*.tsx`
- `electron/**/*.ts`
- `docs/**/*.md`

Но с исключениями для файлов, где старые повреждённые строки могут быть специально оставлены для совместимости или миграции.

Скрипт должен падать, если находит подозрительные паттерны в обычном UI-коде:

```text
Рџ
РЎ
Рњ
Рќ
Рґ
Рё
Р»Р
СЊ
вЂ
```

Для найденных совпадений выводить:

- файл
- номер строки
- кусок строки
- подсказку: "Похоже на сломанную кодировку UTF-8/Windows-1251"

### 5. Подключить guard к проверкам

Добавить команду в `package.json`:

```json
"check:mojibake": "node scripts/check-mojibake.cjs"
```

И подключить её минимум в:

- `npm run type` или отдельный общий check перед пушем
- `scripts/precommit.cjs`
- release gate, если этот файл участвует в сборке релиза

Главное: проверка должна срабатывать до коммита и до локального deploy.

### 6. Зафиксировать правило для будущих правок

В `docs/UI_DESIGN_GUIDE.md` или отдельном инженерном гайде добавить правило:

- не редактировать русские UI-строки через команды, которые могут менять кодировку
- не писать файлы через PowerShell без явного UTF-8
- для ручных правок использовать `apply_patch`
- для массовых преобразований использовать Node с `fs.readFileSync(..., 'utf8')` и `fs.writeFileSync(..., 'utf8')`
- после любых массовых текстовых правок запускать `npm run check:mojibake`

### 7. Проверить в приложении

После исправления нужно проверить вручную:

1. Открыть настройки проекта из шестерёнки в "Мои проекты".
2. Проверить весь текст сверху вниз.
3. Проверить placeholder в заметках и ярлыках.
4. Проверить кнопки сохранения, закрытия, архива, удаления, очистки и копирования.
5. Переключить светлую и тёмную тему.
6. Проверить, что font-family остался в стиле Verstak, а не вернулся жирный моно-шрифт.
7. Проверить, что проектные данные не изменились.

## Минимальный безопасный фикс

Если нужно быстро починить только текущую проблему:

1. Восстановить все mojibake-строки в `src/components/ProjectSettings.tsx`.
2. Запустить `npm.cmd run type`.
3. Запустить `npm.cmd run build`.
4. Проверить окно настроек проекта вручную.

Но это не защищает от повторения.

## Правильный фикс

Сделать полный набор:

1. Восстановить строки.
2. Перенести статичный текст в i18n.
3. Добавить `check:mojibake`.
4. Подключить guard к precommit/check.
5. Обновить инженерное правило по кодировкам.

Так проблема будет решена не только в этом окне, но и предотвращена для будущих правок.

## Риски

- Автоматическое восстановление может испортить редкие строки, если применить его ко всему файлу без фильтра.
- Вынесение в i18n может зацепить много JSX-разметки, поэтому лучше делать его отдельным аккуратным шагом после первичного восстановления текста.
- Некоторые старые mojibake-строки в storage/journal могут быть нужны как fallback для старых данных. Их нельзя массово удалять без проверки.

## Рекомендуемый порядок реализации

1. Сначала быстрый фикс текста в `ProjectSettings.tsx`.
2. Затем guard `check:mojibake`.
3. Затем перенос текстов в i18n.
4. Затем ручная проверка окна проекта.
5. Только после этого локальный deploy и push.
